OOP Pillars 面向对象程序设计基础

The four fundamental concepts that distinguish object-oriented programming from other paradigms. 区分面向对象程序设计与其他编程范式的四个基本概念。

Abstraction抽象

A model that reflects specific aspects of real-world objects or phenomena, capturing relevant details while ignoring everything else. Like how a flight simulator and a booking app model an Airplane differently. 反映真实世界对象或现象中特定内容的模型，能高精度地反映相关详细信息，同时忽略其他内容。就像飞行模拟器和订票应用会以不同方式建模飞机。

Encapsulation封装

The ability of an object to hide parts of its state and behavior from other objects, exposing only a limited interface. Like starting a car with a key — the engine internals are hidden behind a simple interface. 对象对其他对象隐藏其部分状态和行为、仅向程序其他部分暴露有限接口的能力。就像用钥匙启动汽车——发动机内部被隐藏在简单的接口之后。

Inheritance继承

The ability to build new classes on top of existing ones. The main benefit is code reuse — subclasses inherit the parent's interface and implementation, and can extend them with new functionality. 在已有类的基础上创建新类的能力。最主要的好处是代码复用——子类继承父类的接口和实现，并可以通过新功能对其进行扩展。

Polymorphism多态

The ability of a program to detect the actual class of an object and call its implementation, even when the real type is unknown in the current context. An object can "pretend" to be something else. 程序能检测对象所属的实际类，并在当前上下文不知道其真实类型的情况下调用其实现的能力。对象可以"假扮"为其他东西。

Design Principles设计原则

SOLID Principles SOLID 原则

Five core principles of object-oriented design that make software more understandable, flexible, and maintainable. 面向对象设计的五条核心原则，使软件更易理解、更灵活、更易维护。

S

Single Responsibility单一职责原则

A class should have just one reason to change. Each class handles one specific part of the functionality. This reduces complexity and makes the class easier to understand and maintain. 一个类应该只有一个引起变化的原因。每个类只负责功能的一个特定部分，这降低了复杂性，使类更容易理解和维护。

O

Open/Closed开闭原则

Classes should be open for extension but closed for modification. You should be able to add new behavior without changing existing code, typically through interfaces and abstract classes. 类应该对扩展开放，对修改关闭。你应该能够在不修改已有代码的情况下添加新行为，通常通过接口和抽象类来实现。

L

Liskov Substitution里氏替换原则

Subclasses should be substitutable for their base classes without breaking the program. When overriding a method, extend the base behavior rather than replacing it entirely. 子类应该能够替换其基类而不会破坏程序。在重写方法时，应该扩展基类行为，而非完全替换它。

I

Interface Segregation接口隔离原则

Clients should not be forced to depend on interfaces they don't use. Break large interfaces into smaller, more specific ones so that clients only need to know about relevant methods. 客户端不应该被迫依赖它们不使用的接口。将大接口拆分为更小、更具体的接口，使客户端只需了解相关的方法。

D

Dependency Inversion依赖倒置原则

High-level classes should not depend on low-level classes. Both should depend on abstractions. Abstractions should not depend on details — details should depend on abstractions. 高层类不应该依赖低层类，两者都应该依赖于抽象。抽象不应该依赖于细节——细节应该依赖于抽象。

Creational创建型

Object Creation Mechanisms对象创建机制

Factory Method
aka Virtual Constructor工厂方法 / 虚拟构造函数

Defines an interface for creating objects, but lets subclasses decide which class to instantiate. Delegates the instantiation logic to child classes. 定义一个创建对象的接口，但让子类决定实例化哪个类。将实例化逻辑委派给子类。

Creational创建型

When to use适用场景

Use when you don't know beforehand the exact types and dependencies of the objects your code should work with. Also useful when you want to provide users of your library or framework with a way to extend its internal components.当你事先不知道代码应该使用的对象的确切类型和依赖关系时使用。当你想为库或框架的用户提供扩展其内部组件的方式时也很有用。

Structure结构

interface Product { ... }

abstract class Creator {
  abstract createProduct(): Product

  operation() {
    const product = this.createProduct()
    // use product...
  }
}

class ConcreteCreator extends Creator {
  createProduct(): Product {
    return new ConcreteProduct()
  }
}
Abstract Factory
Family of related objects抽象工厂 / 相关对象族

Creates families of related objects without specifying their concrete classes. Ensures that the created products are compatible with each other. 创建一系列相关的对象，而无需指定其具体类。确保创建的产品相互兼容。

Creational创建型

When to use适用场景

Use when your code needs to work with various families of related products, but you don't want it to depend on concrete classes — they might be unknown beforehand or you want to allow for future extensibility.当你的代码需要与多组相关产品交互，但又不希望依赖具体类时使用——它们可能事先未知，或者你希望为未来的可扩展性留出空间。

Builder
Step-by-step construction生成器 / 分步构造

Constructs complex objects step by step. Allows you to produce different types and representations of an object using the same construction process. 分步骤构造复杂对象。允许你使用相同的构造过程生成不同类型和形式的对象。

Creational创建型

When to use适用场景

Use when constructing objects that require many steps. Especially useful when you want to avoid a "telescoping constructor" with too many parameters.当需要通过许多步骤来构建对象时使用。当你想避免拥有太多参数的"伸缩构造函数"时尤其有用。

Prototype
Clone existing objects原型 / 克隆现有对象

Creates new objects by copying an existing object (prototype), delegating the cloning process to the actual objects that are being cloned. 通过复制现有对象（原型）来创建新对象，将克隆过程委托给被克隆的实际对象。

Creational创建型

When to use适用场景

Use when your code shouldn't depend on the concrete classes of objects you need to copy, or when you want to reduce the number of subclasses that only differ in initialization.当代码不应依赖于需要复制的对象的具体类时使用，或当你想减少仅在初始化方式上有所不同的子类数量时使用。

Singleton
Single instance only单例 / 唯一实例

Ensures a class has only one instance while providing a global access point to it. Controls shared resources like a database connection or file system. 确保一个类只有一个实例，同时提供全局访问点。控制共享资源，如数据库连接或文件系统。

Creational创建型

When to use适用场景

Use when a class should have only a single instance available to all clients, such as a shared database object. Also useful when you need stricter control over global variables.当一个类应该只有一个可供所有客户端使用的实例时使用，例如共享的数据库对象。当需要更严格地控制全局变量时也很有用。

Structural结构型

Object Composition对象组合

Adapter
aka Wrapper适配器 / 包装器

Allows objects with incompatible interfaces to collaborate by wrapping one object to make it compatible with another. 通过包装对象使不兼容的接口能够相互协作。

Structural结构型
Bridge
Separate abstraction from implementation桥接 / 分离抽象与实现

Splits a large class or a set of closely related classes into two separate hierarchies — abstraction and implementation — which can be developed independently. 将一个大类或一组紧密相关的类拆分为抽象和实现两个独立的层次结构，使它们可以独立开发。

Structural结构型
Composite
Tree structure of objects组合 / 对象树

Composes objects into tree structures to represent part-whole hierarchies. Lets clients treat individual objects and compositions uniformly. 将对象组合成树形结构来表示"部分-整体"层次关系。让客户端统一处理单个对象和组合对象。

Structural结构型
Decorator
aka Wrapper装饰 / 包装器

Attaches new behaviors to objects by placing them inside wrapper objects that contain the behaviors. A flexible alternative to subclassing. 通过将对象放入包含行为的包装器对象中来为其附加新行为。是子类化的灵活替代方案。

Structural结构型
Facade
Simplified interface外观 / 简化接口

Provides a simplified interface to a library, framework, or any other complex set of classes, hiding the complexity behind a single class. 为库、框架或其他复杂的类集合提供简化的接口，将复杂性隐藏在单个类之后。

Structural结构型
Flyweight
Shared state享元 / 共享状态

Fits more objects into available RAM by sharing common parts of state between multiple objects instead of keeping all data in each object. 通过在多个对象间共享状态的公共部分，而非在每个对象中保存所有数据，使更多对象能装入可用内存。

Structural结构型
Proxy
Access control代理 / 访问控制

Provides a substitute or placeholder for another object. A proxy controls access to the original object, performing something before or after the request reaches it. 为另一个对象提供替代品或占位符。代理控制对原始对象的访问，在请求到达前后执行某些操作。

Structural结构型
Behavioral行为型

Communication & Responsibility沟通与职责

Chain of Responsibility
Pass along a chain责任链 / 沿链传递

Passes requests along a chain of handlers. Each handler decides either to process the request or to pass it to the next handler in the chain. 将请求沿着处理者链传递。每个处理者决定是处理请求还是将其传递给链中的下一个处理者。

Behavioral行为型
Command
Request as object命令 / 请求对象化

Turns a request into a stand-alone object containing all information about the request. Supports undoable operations, queuing, and logging. 将请求转换为包含所有请求信息的独立对象。支持可撤销操作、排队和日志记录。

Behavioral行为型
Iterator
Sequential access迭代器 / 顺序访问

Traverses elements of a collection without exposing its underlying representation (list, stack, tree, etc.). 在不暴露集合底层表现形式（列表、栈、树等）的情况下遍历集合中的所有元素。

Behavioral行为型
Mediator
Central communication hub中介者 / 通信中枢

Reduces chaotic dependencies between objects by restricting direct communications and forcing them to collaborate via a mediator object. 通过限制对象间的直接通信并强制它们通过中介者对象协作，减少对象间的混乱依赖。

Behavioral行为型
Memento
Snapshot / Undo备忘录 / 快照

Saves and restores the previous state of an object without revealing the details of its implementation. Enables undo/redo functionality. 在不暴露对象实现细节的情况下保存和恢复对象之前的状态。支持撤销/重做功能。

Behavioral行为型
Observer
Event subscription观察者 / 事件订阅

Defines a subscription mechanism to notify multiple objects about any events that happen to the observed object. 定义一种订阅机制，以便在被观察对象发生事件时通知多个其他对象。

Behavioral行为型
State
Object behavior changes with state状态 / 行为随状态改变

Lets an object alter its behavior when its internal state changes. It appears as if the object changed its class. 当对象的内部状态改变时改变其行为，看起来就像对象改变了其类。

Behavioral行为型
Strategy
Interchangeable algorithms策略 / 可互换算法

Defines a family of algorithms, encapsulates each one, and makes them interchangeable. Strategy lets the algorithm vary independently from clients that use it. 定义一系列算法，将每个算法封装起来，并使它们可以互换。策略模式让算法独立于使用它的客户端而变化。

Behavioral行为型
Template Method
Algorithm skeleton模板方法 / 算法骨架

Defines the skeleton of an algorithm in a base class, letting subclasses override specific steps without changing the algorithm's overall structure. 在基类中定义算法的骨架，允许子类在不改变算法整体结构的情况下重写特定步骤。

Behavioral行为型
Visitor
Separate algorithms from objects访问者 / 分离算法与对象

Separates algorithms from the objects on which they operate, by moving each algorithm into a separate class called a visitor. 将算法与其所作用的对象分离，把每个算法移入称为"访问者"的独立类中。
