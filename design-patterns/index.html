Design patterns are typical solutions to commonly occurring problems in software design. They are like pre-made blueprints that you can customize to solve a recurring design problem in your code. You can't just find a pattern and copy it into your program — a pattern is not a specific piece of code, but a general concept for solving a particular problem. 设计模式是软件设计中常见问题的典型解决方案。它们就像能根据需求进行调整的预制蓝图，可用于解决代码中反复出现的设计问题。你不能像使用现成的函数或库一样直接在程序中套用某个设计模式——模式并不是一段特定的代码，而是解决特定问题的一般性概念。

1. OOP Foundations 1. 面向对象程序设计基础

Object-Oriented Programming (OOP) is a paradigm based on the concept of wrapping pieces of data, and behavior related to that data, into special bundles called objects, which are constructed from blueprints called classes. Four pillars distinguish OOP from other paradigms: 面向对象程序设计 (OOP) 是一种范式，其基本理念是将数据块及与数据相关的行为封装成特殊的实体——对象，而对象的生成工作基于程序员给出的"蓝图"——类。四个基本概念使其区别于其他编程范式：

Abstraction抽象

A model that reflects specific aspects of real-world objects or phenomena, capturing relevant details while ignoring the rest. A flight simulator and a booking app both model an Airplane, but with completely different attributes and behaviors. 反映真实世界对象或现象中特定内容的模型，能高精度地反映相关详细信息，同时忽略其他内容。飞行模拟器和订票应用都会建模 Airplane，但属性和行为完全不同。

Encapsulation封装

The ability of an object to hide parts of its state and behavior from other objects, exposing only a limited interface to the rest of the program. Like starting a car with a key — the engine internals are hidden behind a simple interface. 对象对其他对象隐藏其部分状态和行为、仅向程序其他部分暴露有限接口的能力。就像用钥匙启动汽车——发动机内部被隐藏在简单的接口之后。

Inheritance继承

The ability to build new classes on top of existing ones. The main benefit is code reuse. Subclasses inherit the parent's interface and implementation, and can extend them with additional functionality. A Cat and a Dog can both inherit from Animal. 在已有类的基础上创建新类的能力。最主要的好处是代码复用。子类继承父类的接口和实现，并可对其进行扩展。Cat 和 Dog 都可以继承自 Animal。

Polymorphism多态

The ability of a program to detect the actual class of an object and call its implementation, even when its real type is unknown in the current context. An object can "pretend" to be something it extends or implements. 程序能检测对象所属的实际类、并在当前上下文不知道其真实类型的情况下调用其实现的能力。对象可以"假扮"为其扩展或实现的东西。 

bag = [new Cat(), new Dog()]

foreach (Animal a : bag)
  a.makeSound()
// Meow!
// Woof!

2. Object Relationships 2. 对象之间的关系

Relations between objects and classes, ordered from weakest to strongest coupling: 对象和类之间的关系，按耦合程度从弱到强排列：

Dependency依赖 → Association关联 → Aggregation聚合 → Composition组合 → Inheritance继承
  • Dependency依赖 — Changes to class B may affect class A. The weakest relation. 对类 B 的修改会影响到类 A。最弱的关系。
  • Association关联 — Object A knows about object B. A special kind of dependency where there is a permanent link. 对象 A 知道对象 B。一种存在永久性联系的特殊依赖。
  • Aggregation聚合 — "Has-a" relationship. Components can exist independently of the container. "拥有"关系。组件可以独立于容器存在。
  • Composition组合 — Components can only exist as part of the container; it manages their lifecycle. 组件仅能作为容器的一部分存在，容器管理其生命周期。
  • Inheritance继承 — Class A inherits interface and implementation of class B. The strongest coupling. 类 A 继承类 B 的接口和实现。最强的耦合。

3. SOLID Principles 3. SOLID 原则

Five core principles of object-oriented design that make software more understandable, flexible, and maintainable. 面向对象设计的五条核心原则，使软件更易理解、更灵活、更易维护。

S

Single Responsibility Principle单一职责原则

A class should have just one reason to change. If a class handles too many things, you have to change it every time one of those things changes — risking breaking the others. 一个类应该只有一个引起变化的原因。如果一个类负责太多事情，每当其中一项发生变化时你就必须修改它——可能会破坏其他部分。

O

Open/Closed Principle开闭原则

Classes should be open for extension but closed for modification. Add new behavior without changing existing code, typically through interfaces and abstract classes. 类应该对扩展开放，对修改关闭。在不修改已有代码的情况下添加新行为，通常通过接口和抽象类来实现。

L

Liskov Substitution Principle里氏替换原则

Subclasses should be substitutable for their base classes. When overriding a method, extend the base behavior rather than replacing it with something entirely different. 子类应该能够替换其基类。在重写方法时，应该扩展基类行为，而非用完全不同的东西替换它。

I

Interface Segregation Principle接口隔离原则

Clients should not be forced to depend on interfaces they do not use. Break "fat" interfaces into smaller, more specific ones. 客户端不应该被迫依赖它们不使用的接口。将"胖"接口拆分为更小、更具体的接口。

D

Dependency Inversion Principle依赖倒置原则

High-level classes should not depend on low-level classes. Both should depend on abstractions. Details should depend on abstractions, not the other way around. 高层类不应该依赖低层类，两者都应该依赖于抽象。细节应该依赖于抽象，而非反过来。

4. Creational Patterns 4. 创建型模式

Creational patterns provide object creation mechanisms that increase flexibility and reuse of existing code. 创建型模式提供对象创建机制，增加已有代码的灵活性和可复用性。

Creational创建型 Factory Method

aka Virtual Constructor — 工厂方法工厂方法 / 虚拟构造函数

Defines an interface for creating objects, but lets subclasses decide which class to instantiate. Use when you don't know beforehand the exact types your code should work with, or when you want to let users extend your library's internal components. 定义一个创建对象的接口，但让子类决定实例化哪个类。当你事先不知道代码应该使用的确切类型时，或当你想让用户能够扩展库的内部组件时使用。

Creational创建型 Abstract Factory

Families of related objects — 抽象工厂抽象工厂 / 相关对象族

Creates families of related objects without specifying their concrete classes. Ensures that created products are compatible with each other. Use when your code needs to work with various families of related products. 创建一系列相关的对象，而无需指定其具体类。确保创建的产品相互兼容。当你的代码需要与多组相关产品交互时使用。

Creational创建型 Builder

Step-by-step construction — 生成器生成器 / 分步构造

Constructs complex objects step by step. Allows you to produce different types and representations using the same construction process. Especially useful to avoid a "telescoping constructor" with too many parameters. 分步骤构造复杂对象。允许你使用相同的构造过程生成不同类型和形式的对象。当你想避免拥有太多参数的"伸缩构造函数"时尤其有用。

Creational创建型 Prototype

Clone existing objects — 原型原型 / 克隆现有对象

Creates new objects by copying an existing object (prototype), delegating the cloning process to the actual objects being cloned. Use when your code shouldn't depend on the concrete classes of objects you need to copy. 通过复制现有对象（原型）来创建新对象，将克隆过程委托给被克隆的实际对象。当代码不应依赖于需要复制的对象的具体类时使用。

Creational创建型 Singleton

Single instance only — 单例单例 / 唯一实例

Ensures a class has only one instance while providing a global access point to it. Controls shared resources like a database connection. Use when a class should have only a single instance available to all clients. 确保一个类只有一个实例，同时提供全局访问点。控制共享资源，如数据库连接。当一个类应该只有一个可供所有客户端使用的实例时使用。

Tip: The Factory Method is a good starting point. If your code becomes complex with many factories, consider moving to Abstract Factory, Prototype, or Builder. 提示：工厂方法是一个很好的起点。如果你的代码因大量工厂而变得复杂，可以考虑转向抽象工厂、原型或生成器。

5. Structural Patterns 5. 结构型模式

Structural patterns explain how to assemble objects and classes into larger structures while keeping them flexible and efficient. 结构型模式介绍如何将对象和类组装成较大的结构，同时保持结构的灵活和高效。

Structural结构型 Adapter

aka Wrapper — 适配器适配器 / 包装器

Allows objects with incompatible interfaces to collaborate by wrapping one object to make it compatible with another. A "translator" between two incompatible interfaces. 通过包装对象使不兼容的接口能够相互协作。是两个不兼容接口之间的"翻译者"。

Structural结构型 Bridge

Separate abstraction from implementation — 桥接桥接 / 分离抽象与实现

Splits a large class or a set of closely related classes into two separate hierarchies — abstraction and implementation — which can be developed independently of each other. 将一个大类或一组紧密相关的类拆分为抽象和实现两个独立的层次结构，使它们可以独立开发。

Structural结构型 Composite

Tree structure — 组合组合 / 对象树

Composes objects into tree structures to represent part-whole hierarchies. Lets clients treat individual objects and compositions uniformly through a common interface. 将对象组合成树形结构来表示"部分-整体"层次关系。让客户端通过公共接口统一处理单个对象和组合对象。

Structural结构型 Decorator

aka Wrapper — 装饰装饰 / 包装器

Attaches new behaviors to objects by placing them inside wrapper objects that contain the behaviors. A flexible alternative to subclassing for extending functionality. 通过将对象放入包含行为的包装器对象中来附加新行为。是通过子类化扩展功能的灵活替代方案。

Structural结构型 Facade

Simplified interface — 外观外观 / 简化接口

Provides a simplified interface to a library, framework, or complex set of classes. Hides system complexity behind a single, easy-to-use class. 为库、框架或复杂的类集合提供简化的接口。将系统复杂性隐藏在单个易用的类之后。

Structural结构型 Flyweight

Shared state — 享元享元 / 共享状态

Fits more objects into available RAM by sharing common parts of state between multiple objects instead of keeping all data in each object. 通过在多个对象间共享状态的公共部分，而非在每个对象中保存所有数据，使更多对象能装入可用内存。

Structural结构型 Proxy

Access control — 代理代理 / 访问控制

Provides a substitute or placeholder for another object. Controls access to the original, performing something before or after the request reaches it (lazy loading, caching, access control, logging). 为另一个对象提供替代品或占位符。控制对原始对象的访问，在请求到达前后执行某些操作（延迟加载、缓存、访问控制、日志记录）。

6. Behavioral Patterns 6. 行为型模式

Behavioral patterns are concerned with algorithms and the assignment of responsibilities between objects. They describe not just patterns of objects or classes but also the patterns of communication between them. 行为型模式负责对象间的高效沟通和职责委派。它们不仅描述对象或类的模式，还描述它们之间的通信模式。

Behavioral行为型 Chain of Responsibility

Pass along a chain — 责任链责任链 / 沿链传递

Passes requests along a chain of handlers. Each handler decides either to process the request or to pass it to the next handler in the chain. Decouples senders from receivers. 将请求沿着处理者链传递。每个处理者决定是处理请求还是将其传递给链中的下一个处理者。将发送者与接收者解耦。

Behavioral行为型 Command

Request as object — 命令命令 / 请求对象化

Turns a request into a stand-alone object containing all information about the request. Supports undoable operations, queuing, and logging of requests. 将请求转换为包含所有请求信息的独立对象。支持可撤销操作、排队和请求日志记录。

Behavioral行为型 Iterator

Sequential access — 迭代器迭代器 / 顺序访问

Traverses elements of a collection without exposing its underlying representation (list, stack, tree, etc.). Provides a uniform way to access elements regardless of data structure. 在不暴露集合底层表现形式（列表、栈、树等）的情况下遍历集合中的所有元素。提供统一的元素访问方式，与数据结构无关。

Behavioral行为型 Mediator

Central hub — 中介者中介者 / 通信中枢

Reduces chaotic dependencies between objects by restricting direct communications and forcing them to collaborate only via a mediator object. Components become independent of each other. 通过限制对象间的直接通信并强制它们仅通过中介者对象协作，减少对象间的混乱依赖。组件变得彼此独立。

Behavioral行为型 Memento

Snapshot / Undo — 备忘录备忘录 / 快照

Saves and restores the previous state of an object without revealing the details of its implementation. Enables undo/redo functionality without violating encapsulation. 在不暴露对象实现细节的情况下保存和恢复对象之前的状态。在不违反封装的前提下实现撤销/重做功能。

Behavioral行为型 Observer

Event subscription — 观察者观察者 / 事件订阅

Defines a subscription mechanism to notify multiple objects about any events that happen to the object they're observing. The basis for the event-driven architecture. 定义一种订阅机制，以便在被观察对象发生事件时通知多个其他对象。事件驱动架构的基础。

Behavioral行为型 State

Behavior changes with state — 状态状态 / 行为随状态改变

Lets an object alter its behavior when its internal state changes. It appears as if the object changed its class. An alternative to large state-based conditionals. 当对象的内部状态改变时改变其行为，看起来就像对象改变了其类。是大型基于状态的条件语句的替代方案。

Behavioral行为型 Strategy

Interchangeable algorithms — 策略策略 / 可互换算法

Defines a family of algorithms, encapsulates each one, and makes them interchangeable. Lets the algorithm vary independently from clients that use it. 定义一系列算法，将每个算法封装起来，并使它们可以互换。让算法独立于使用它的客户端而变化。

Behavioral行为型 Template Method

Algorithm skeleton — 模板方法模板方法 / 算法骨架

Defines the skeleton of an algorithm in a base class, letting subclasses override specific steps without changing the algorithm's overall structure. 在基类中定义算法的骨架，允许子类在不改变算法整体结构的情况下重写特定步骤。

Behavioral行为型 Visitor

Separate algorithms from objects — 访问者访问者 / 分离算法与对象

Separates algorithms from the objects on which they operate by moving each algorithm into a separate class called a visitor. Lets you add new operations without modifying existing classes. 将算法与其所作用的对象分离，把每个算法移入称为"访问者"的独立类中。让你在不修改现有类的情况下添加新操作。

Note: Patterns are often used together. For example, many designs start with Factory Method and evolve toward Abstract Factory, Prototype, or Builder. Strategy can be combined with Bridge, and State can be considered a special case of Strategy. 注意：模式常常组合使用。例如，许多设计从工厂方法开始，逐步演变为抽象工厂、原型或生成器。策略可以与桥接组合使用，而状态可以被视为策略的特殊情况。
