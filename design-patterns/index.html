Design patterns are typical solutions to commonly occurring problems in software design. They are like pre-made blueprints that you can customize to solve a recurring design problem in your code. You can't just find a pattern and copy it into your program — a pattern is not a specific piece of code, but a general concept for solving a particular problem. 设计模式是软件设计中常见问题的典型解决方案。它们就像能根据需求进行调整的预制蓝图，可用于解决代码中反复出现的设计问题。你不能像使用现成的函数或库一样直接在程序中套用某个设计模式——模式并不是一段特定的代码，而是解决特定问题的一般性概念。

1. OOP Foundations 1. 面向对象程序设计基础

Object-Oriented Programming (OOP) is a paradigm based on the concept of wrapping pieces of data, and behavior related to that data, into special bundles called objects, which are constructed from blueprints called classes. Four pillars distinguish OOP from other paradigms: 面向对象程序设计 (OOP) 是一种范式，其基本理念是将数据块及与数据相关的行为封装成特殊的实体——对象，而对象的生成工作基于程序员给出的"蓝图"——类。四个基本概念使其区别于其他编程范式：

Abstraction抽象

A model that reflects specific aspects of real-world objects or phenomena, capturing relevant details while ignoring the rest. A flight simulator and a booking app both model an Airplane, but with completely different attributes and behaviors. 反映真实世界对象或现象中特定内容的模型，能高精度地反映相关详细信息，同时忽略其他内容。飞行模拟器和订票应用都会建模 Airplane，但属性和行为完全不同。

Encapsulation封装

The ability of an object to hide parts of its state and behavior from other objects, exposing only a limited interface to the rest of the program. Like starting a car with a key — the engine internals are hidden behind a simple interface. 对象对其他对象隐藏其部分状态和行为、仅向程序其他部分暴露有限接口的能力。就像用钥匙启动汽车——发动机内部被隐藏在简单的接口之后。

Inheritance继承

The ability to build new classes on top of existing ones. The main benefit is code reuse. Subclasses inherit the parent's interface and implementation, and can extend them with additional functionality. A Cat and a Dog can both inherit from Animal. 在已有类的基础上创建新类的能力。最主要的好处是代码复用。子类继承父类的接口和实现，并可对其进行扩展。Cat 和 Dog 都可以继承自 Animal。

Polymorphism多态

The ability of a program to detect the actual class of an object and call its implementation, even when its real type is unknown in the current context. An object can "pretend" to be something it extends or implements. 程序能检测对象所属的实际类、并在当前上下文不知道其真实类型的情况下调用其实现的能力。对象可以"假扮"为其扩展或实现的东西。 

bag = [new Cat(), new Dog()]

foreach (Animal a : bag)
  a.makeSound()
// Meow!
// Woof!

2. Object Relationships 2. 对象之间的关系

Relations between objects and classes, ordered from weakest to strongest coupling: 对象和类之间的关系，按耦合程度从弱到强排列：

Dependency依赖 → Association关联 → Aggregation聚合 → Composition组合 → Inheritance继承
  • Dependency依赖 — Changes to class B may affect class A. The weakest relation. 对类 B 的修改会影响到类 A。最弱的关系。
  • Association关联 — Object A knows about object B. A special kind of dependency where there is a permanent link. 对象 A 知道对象 B。一种存在永久性联系的特殊依赖。
  • Aggregation聚合 — "Has-a" relationship. Components can exist independently of the container. "拥有"关系。组件可以独立于容器存在。
  • Composition组合 — Components can only exist as part of the container; it manages their lifecycle. 组件仅能作为容器的一部分存在，容器管理其生命周期。
  • Inheritance继承 — Class A inherits interface and implementation of class B. The strongest coupling. 类 A 继承类 B 的接口和实现。最强的耦合。

3. SOLID Principles 3. SOLID 原则

Five core principles of object-oriented design that make software more understandable, flexible, and maintainable. 面向对象设计的五条核心原则，使软件更易理解、更灵活、更易维护。

S

Single Responsibility Principle单一职责原则

A class should have just one reason to change. If a class handles too many things, you have to change it every time one of those things changes — risking breaking the others. 一个类应该只有一个引起变化的原因。如果一个类负责太多事情，每当其中一项发生变化时你就必须修改它——可能会破坏其他部分。

O

Open/Closed Principle开闭原则

Classes should be open for extension but closed for modification. Add new behavior without changing existing code, typically through interfaces and abstract classes. 类应该对扩展开放，对修改关闭。在不修改已有代码的情况下添加新行为，通常通过接口和抽象类来实现。

L

Liskov Substitution Principle里氏替换原则

Subclasses should be substitutable for their base classes. When overriding a method, extend the base behavior rather than replacing it with something entirely different. 子类应该能够替换其基类。在重写方法时，应该扩展基类行为，而非用完全不同的东西替换它。

I

Interface Segregation Principle接口隔离原则

Clients should not be forced to depend on interfaces they do not use. Break "fat" interfaces into smaller, more specific ones. 客户端不应该被迫依赖它们不使用的接口。将"胖"接口拆分为更小、更具体的接口。

D

Dependency Inversion Principle依赖倒置原则

High-level classes should not depend on low-level classes. Both should depend on abstractions. Details should depend on abstractions, not the other way around. 高层类不应该依赖低层类，两者都应该依赖于抽象。细节应该依赖于抽象，而非反过来。

4. Creational Patterns 4. 创建型模式

Creational patterns provide object creation mechanisms that increase flexibility and reuse of existing code. 创建型模式提供对象创建机制，增加已有代码的灵活性和可复用性。

Creational创建型 Factory Method

aka Virtual Constructor — 工厂方法工厂方法 / 虚拟构造函数

Defines an interface for creating objects, but lets subclasses decide which class to instantiate. Use when you don't know beforehand the exact types your code should work with, or when you want to let users extend your library's internal components. 定义一个创建对象的接口，但让子类决定实例化哪个类。当你事先不知道代码应该使用的确切类型时，或当你想让用户能够扩展库的内部组件时使用。

Problem & Solution问题与解决方案
Problem问题

Imagine you're developing a logistics management app. The first version only handles truck transport, so most code lives in the Truck class. Later, sea shipping companies want support too. Adding Ship requires changing all existing code — and every new transport type means another painful rewrite.假设你正在开发一款物流管理应用。最初版本只能处理卡车运输，因此大部分代码都位于 Truck 类中。后来海运公司也想要支持。添加 Ship 类需要修改全部代码——每次新增运输方式都要再次大幅修改。

Solution解决方案

Replace direct constructor calls with a special factory method. Subclasses override this method to return different product types. All products must implement a common interface (Transport with a deliver() method), so client code works with any transport type without knowing the concrete class.使用特殊的工厂方法代替对构造函数的直接调用。子类重写该方法以返回不同类型的产品。所有产品都必须实现共同接口（Transport 及其 deliver() 方法），这样客户端代码无需了解具体类就能使用任何运输方式。

Pseudocode伪代码

Cross-platform dialog example: the base Dialog class uses a factory method to create buttons, so subclasses can return platform-specific buttons.跨平台对话框示例：基础 Dialog 类使用工厂方法创建按钮，子类可返回特定平台的按钮。

class Dialog is
    abstract method createButton(): Button

    method render() is
        Button okButton = createButton()
        okButton.onClick(closeDialog)
        okButton.render()

class WindowsDialog extends Dialog is
    method createButton(): Button is
        return new WindowsButton()

class WebDialog extends Dialog is
    method createButton(): Button is
        return new HTMLButton()

interface Button is
    method render()
    method onClick(f)

class WindowsButton implements Button is
    method render(a, b) is
        // Render a button in Windows style.
    method onClick(f) is
        // Bind an OS click event.

class HTMLButton implements Button is
    method render(a, b) is
        // Return an HTML representation of a button.
    method onClick(f) is
        // Bind a web browser click event.

class Application is
    field dialog: Dialog

    method initialize() is
        config = readApplicationConfigFile()
        if (config.OS == "Windows") then
            dialog = new WindowsDialog()
        else if (config.OS == "Web") then
            dialog = new WebDialog()
        else
            throw new Exception("Unknown OS.")

    method main() is
        this.initialize()
        dialog.render()
Pros & Cons优缺点
  • Avoids tight coupling between creator and concrete products.避免创建者和具体产品之间的紧密耦合。
  • Single Responsibility: product creation code in one place.单一职责原则：产品创建代码集中在一处。
  • Open/Closed: introduce new products without changing existing code.开闭原则：无需更改现有代码即可引入新产品。
  • May require many new subclasses, increasing complexity.可能需要引入许多新的子类，代码因此变得更复杂。

Creational创建型 Abstract Factory

Families of related objects — 抽象工厂抽象工厂 / 相关对象族

Creates families of related objects without specifying their concrete classes. Ensures that created products are compatible with each other. Use when your code needs to work with various families of related products. 创建一系列相关的对象，而无需指定其具体类。确保创建的产品相互兼容。当你的代码需要与多组相关产品交互时使用。

Problem & Solution问题与解决方案
Problem问题

A furniture shop simulator with product families (Chair, Sofa, CoffeeTable) in different styles (Modern, Victorian, ArtDeco). You need to ensure style consistency — a Modern sofa shouldn't be paired with a Victorian chair.一个家具商店模拟器，包含产品族（椅子、沙发、咖啡桌）和不同风格（现代、维多利亚、装饰风艺术）。你需要确保风格一致——现代风沙发不应与维多利亚风椅子搭配。

Solution解决方案

Declare interfaces for each product type, then create a factory interface with methods like createChair(), createSofa(). Each concrete factory (ModernFurnitureFactory, VictorianFurnitureFactory) only creates matching products.为每种产品类型声明接口，然后创建一个包含 createChair()、createSofa() 等方法的工厂接口。每个具体工厂（ModernFurnitureFactory、VictorianFurnitureFactory）只创建匹配的产品。

Pseudocode伪代码

Cross-platform UI elements: each factory creates matching buttons and checkboxes for a specific OS.跨平台 UI 元素：每个工厂为特定操作系统创建匹配的按钮和复选框。

interface GUIFactory is
    method createButton(): Button
    method createCheckbox(): Checkbox

class WinFactory implements GUIFactory is
    method createButton(): Button is
        return new WinButton()
    method createCheckbox(): Checkbox is
        return new WinCheckbox()

class MacFactory implements GUIFactory is
    method createButton(): Button is
        return new MacButton()
    method createCheckbox(): Checkbox is
        return new MacCheckbox()

class Application is
    private field factory: GUIFactory
    private field button: Button
    constructor Application(factory: GUIFactory) is
        this.factory = factory
    method createUI() is
        this.button = factory.createButton()
    method paint() is
        button.paint()

// App selects factory at startup based on config
class ApplicationConfigurator is
    method main() is
        config = readApplicationConfigFile()
        if (config.OS == "Windows") then
            factory = new WinFactory()
        else if (config.OS == "Mac") then
            factory = new MacFactory()
        Application app = new Application(factory)
Pros & Cons优缺点
  • Products from the same factory are guaranteed compatible.同一工厂生成的产品相互匹配。
  • Avoids coupling between client code and concrete products.避免客户端和具体产品代码的耦合。
  • Single Responsibility & Open/Closed principles.单一职责原则和开闭原则。
  • Introduces many new interfaces and classes — increased complexity.需要引入众多接口和类，代码可能更加复杂。

Creational创建型 Builder

Step-by-step construction — 生成器生成器 / 分步构造

Constructs complex objects step by step. Allows you to produce different types and representations using the same construction process. Especially useful to avoid a "telescoping constructor" with too many parameters. 分步骤构造复杂对象。允许你使用相同的构造过程生成不同类型和形式的对象。当你想避免拥有太多参数的"伸缩构造函数"时尤其有用。

Pseudocode伪代码

Building cars and their manuals using the same construction steps. A Director knows the build order; different builders produce different outputs.使用相同的构造步骤建造汽车及其使用手册。Director 知道构建顺序；不同的生成器产生不同的输出。

interface Builder is
    method reset()
    method setSeats(...)
    method setEngine(...)
    method setTripComputer(...)
    method setGPS(...)

class CarBuilder implements Builder is
    private field car: Car
    constructor CarBuilder() is
        this.reset()
    method reset() is
        this.car = new Car()
    method setSeats(...) is // Set car seat count
    method setEngine(...) is // Install engine
    method setTripComputer(...) is // Install trip computer
    method setGPS(...) is // Install GPS
    method getProduct(): Car is
        product = this.car
        this.reset()
        return product

class Director is
    method constructSportsCar(builder: Builder) is
        builder.reset()
        builder.setSeats(2)
        builder.setEngine(new SportEngine())
        builder.setTripComputer(true)
        builder.setGPS(true)

class Application is
    method makeCar() is
        director = new Director()
        CarBuilder builder = new CarBuilder()
        director.constructSportsCar(builder)
        Car car = builder.getProduct()
Pros & Cons优缺点
  • Construct objects step-by-step, defer or recurse steps.分步创建对象，可暂缓或递归执行步骤。
  • Reuse same construction code for different product representations.生成不同形式的产品时可复用相同的制造代码。
  • Single Responsibility: isolate complex construction from business logic.单一职责原则：将复杂构造代码与业务逻辑分离。
  • Overall code complexity increases due to new classes.需要新增多个类，代码整体复杂程度增加。

Creational创建型 Prototype

Clone existing objects — 原型原型 / 克隆现有对象

Creates new objects by copying an existing object (prototype), delegating the cloning process to the actual objects being cloned. Use when your code shouldn't depend on the concrete classes of objects you need to copy. 通过复制现有对象（原型）来创建新对象，将克隆过程委托给被克隆的实际对象。当代码不应依赖于需要复制的对象的具体类时使用。

Pseudocode伪代码

Cloning geometric shapes. Each shape implements clone() via a copy constructor that copies all fields including private ones.克隆几何图形。每个形状通过复制构造函数实现 clone()，复制所有字段包括私有字段。

abstract class Shape is
    field X: int
    field Y: int
    field color: string

    constructor Shape(source: Shape) is
        this.X = source.X
        this.Y = source.Y
        this.color = source.color

    abstract method clone(): Shape

class Rectangle extends Shape is
    field width: int
    field height: int

    constructor Rectangle(source: Rectangle) is
        super(source)
        this.width = source.width
        this.height = source.height

    method clone(): Shape is
        return new Rectangle(this)

class Circle extends Shape is
    field radius: int

    constructor Circle(source: Circle) is
        super(source)
        this.radius = source.radius

    method clone(): Shape is
        return new Circle(this)

// Client code
shapes = [new Circle(), new Rectangle()]
shapesCopy = []
foreach (s in shapes) do
    shapesCopy.add(s.clone())
Pros & Cons优缺点
  • Clone objects without coupling to their concrete classes.克隆对象而无需与具体类耦合。
  • Avoid repeated initialization code; clone pre-configured prototypes.避免重复的初始化代码；克隆预配置的原型。
  • Produce complex objects more conveniently.更方便地生成复杂对象。
  • Cloning objects with circular references can be tricky.克隆包含循环引用的复杂对象可能很麻烦。

Creational创建型 Singleton

Single instance only — 单例单例 / 唯一实例

Ensures a class has only one instance while providing a global access point to it. Controls shared resources like a database connection. Use when a class should have only a single instance available to all clients. 确保一个类只有一个实例，同时提供全局访问点。控制共享资源，如数据库连接。当一个类应该只有一个可供所有客户端使用的实例时使用。

Pseudocode伪代码

A Database class that ensures only one connection instance exists. Uses lazy initialization with thread-safe locking.一个确保只有一个连接实例的 Database 类。使用延迟初始化和线程安全的锁机制。

class Database is
    private static field instance: Database

    private constructor Database() is
        // Initialize database connection...

    public static method getInstance() is
        if (Database.instance == null) then
            acquireThreadLock() and then
                if (Database.instance == null) then
                    Database.instance = new Database()
        return Database.instance

    public method query(sql) is
        // All database queries go through this method.

class Application is
    method main() is
        Database foo = Database.getInstance()
        foo.query("SELECT ...")
        Database bar = Database.getInstance()
        bar.query("SELECT ...")
        // foo and bar contain the same object.
Pros & Cons优缺点
  • Guarantees a class has only one instance.保证一个类只有一个实例。
  • Global access point to that instance.获得指向该实例的全局访问节点。
  • Initialized only on first request (lazy init).仅在首次请求时初始化（延迟初始化）。
  • Violates Single Responsibility — solves two problems at once.违反单一职责原则——同时解决两个问题。
  • Can mask bad design; requires special handling in multithreaded environments.可能掩盖不良设计；多线程环境下需要特殊处理。
  • Difficult to unit test due to private constructor.由于私有构造函数，单元测试较困难。

Tip: The Factory Method is a good starting point. If your code becomes complex with many factories, consider moving to Abstract Factory, Prototype, or Builder. 提示：工厂方法是一个很好的起点。如果你的代码因大量工厂而变得复杂，可以考虑转向抽象工厂、原型或生成器。

5. Structural Patterns 5. 结构型模式

Structural patterns explain how to assemble objects and classes into larger structures while keeping them flexible and efficient. 结构型模式介绍如何将对象和类组装成较大的结构，同时保持结构的灵活和高效。

Structural结构型 Adapter

aka Wrapper — 适配器适配器 / 包装器

Allows objects with incompatible interfaces to collaborate by wrapping one object to make it compatible with another. A "translator" between two incompatible interfaces. 通过包装对象使不兼容的接口能够相互协作。是两个不兼容接口之间的"翻译者"。

Pseudocode伪代码

The classic "square peg in a round hole" example. SquarePegAdapter pretends to be a RoundPeg.经典的"方钉与圆孔"示例。SquarePegAdapter 假扮为 RoundPeg。

class RoundHole is
    constructor RoundHole(radius) { ... }
    method getRadius() is // Return hole radius
    method fits(peg: RoundPeg) is
        return this.getRadius() >= peg.getRadius()

class RoundPeg is
    constructor RoundPeg(radius) { ... }
    method getRadius() is // Return peg radius

class SquarePeg is
    constructor SquarePeg(width) { ... }
    method getWidth() is // Return square peg width

class SquarePegAdapter extends RoundPeg is
    private field peg: SquarePeg

    constructor SquarePegAdapter(peg: SquarePeg) is
        this.peg = peg

    method getRadius() is
        // Calculate minimum circle radius that fits the square peg
        return peg.getWidth() * Math.sqrt(2) / 2

// Client code
hole = new RoundHole(5)
rpeg = new RoundPeg(5)
hole.fits(rpeg) // true

small_sqpeg = new SquarePeg(5)
small_sqpeg_adapter = new SquarePegAdapter(small_sqpeg)
hole.fits(small_sqpeg_adapter) // true
Pros & Cons优缺点
  • Single Responsibility: separate interface conversion from business logic.单一职责原则：将接口转换代码与业务逻辑分离。
  • Open/Closed: add new adapters without modifying existing client code.开闭原则：无需修改现有客户端代码即可添加新适配器。
  • Increases overall code complexity. Sometimes simpler to change the service class directly.代码整体复杂度增加。有时直接修改服务类会更简单。

Structural结构型 Bridge

Separate abstraction from implementation — 桥接桥接 / 分离抽象与实现

Splits a large class or a set of closely related classes into two separate hierarchies — abstraction and implementation — which can be developed independently of each other. 将一个大类或一组紧密相关的类拆分为抽象和实现两个独立的层次结构，使它们可以独立开发。

Pseudocode伪代码

Remote controls (abstraction) and devices (implementation) can vary independently. An AdvancedRemote adds mute without changing any device class.遥控器（抽象）和设备（实现）可以独立变化。AdvancedRemote 可以添加静音功能而不改变任何设备类。

class RemoteControl is
    protected field device: Device

    constructor RemoteControl(device: Device) is
        this.device = device

    method togglePower() is
        if (device.isEnabled()) then
            device.disable()
        else
            device.enable()

    method volumeDown() is
        device.setVolume(device.getVolume() - 10)

    method volumeUp() is
        device.setVolume(device.getVolume() + 10)

class AdvancedRemoteControl extends RemoteControl is
    method mute() is
        device.setVolume(0)

interface Device is
    method isEnabled()
    method enable()
    method disable()
    method getVolume()
    method setVolume(percent)

class Tv implements Device is // ...
class Radio implements Device is // ...

// Client code
tv = new Tv()
remote = new RemoteControl(tv)
remote.togglePower()

radio = new Radio()
remote = new AdvancedRemoteControl(radio)
Pros & Cons优缺点
  • Create platform-independent classes and programs.可以创建与平台无关的类和程序。
  • Open/Closed: new abstractions and implementations don't affect each other.开闭原则：新增抽象和实现部分不会相互影响。
  • Single Responsibility: high-level logic vs. platform details.单一职责原则：高层逻辑与平台细节分离。
  • Can overcomplicate highly cohesive classes.对高内聚的类使用该模式可能会让代码更复杂。

Structural结构型 Composite

Tree structure — 组合组合 / 对象树

Composes objects into tree structures to represent part-whole hierarchies. Lets clients treat individual objects and compositions uniformly through a common interface. 将对象组合成树形结构来表示"部分-整体"层次关系。让客户端通过公共接口统一处理单个对象和组合对象。

Pseudocode伪代码

6. Behavioral Patterns 6. 行为型模式

Behavioral patterns are concerned with algorithms and the assignment of responsibilities between objects. They describe not just patterns of objects or classes but also the patterns of communication between them. 行为型模式负责对象间的高效沟通和职责委派。它们不仅描述对象或类的模式，还描述它们之间的通信模式。

Behavioral行为型 Chain of Responsibility

Pass along a chain — 责任链责任链 / 沿链传递

Passes requests along a chain of handlers. Each handler decides either to process the request or to pass it to the next handler in the chain. Decouples senders from receivers. 将请求沿着处理者链传递。每个处理者决定是处理请求还是将其传递给链中的下一个处理者。将发送者与接收者解耦。

  • Single Responsibility & Open/Closed: decouple senders, add new handlers freely.单一职责和开闭原则：解耦发送者，自由添加新处理者。
  • Some requests may end up unhandled.部分请求可能未被处理。

Turns a request into a stand-alone object containing all information about the request. Supports undoable operations, queuing, and logging of requests. 将请求转换为包含所有请求信息的独立对象。支持可撤销操作、排队和请求日志记录。

Behavioral行为型 Observer

Event subscription — 观察者观察者 / 事件订阅

Defines a subscription mechanism to notify multiple objects about any events that happen to the object they're observing. The basis for the event-driven architecture. 定义一种订阅机制，以便在被观察对象发生事件时通知多个其他对象。事件驱动架构的基础。

Problem & Solution问题与解决方案

Imagine Customer and Store objects. Customers want to know when a product arrives. They could visit daily (wasteful), or the store could spam everyone (annoying). Observer adds a subscription mechanism to the publisher: 1) a list storing subscriber references, 2) methods to add/remove subscribers. When an event occurs, the publisher traverses the list and calls each subscriber's notification method via a common interface.

假设有顾客和商店对象。顾客想知道产品何时到货。他们可以每天去查看（浪费时间），或商店通知所有人（打扰）。观察者模式为发布者添加订阅机制：1）存储订阅者引用的列表，2）添加/删除订阅者的方法。事件发生时，发布者遍历列表并通过通用接口调用每个订阅者的通知方法。

Pseudocode伪代码

A text editor notifies other service objects (logging, email) about state changes via an event manager.

文本编辑器通过事件管理器将状态变化通知给其他服务对象（日志、邮件提醒）。

// Publisher base class with subscription management
class EventManager is
    private field listeners: hash map of event types and listeners

    method subscribe(eventType, listener) is
        listeners.add(eventType, listener)

    method unsubscribe(eventType, listener) is
        listeners.remove(eventType, listener)

    method notify(eventType, data) is
        foreach (listener in listeners.of(eventType)) do
            listener.update(data)

// Concrete publisher
class Editor is
    public field events: EventManager
    private field file: File

    constructor Editor() is
        events = new EventManager()

    method openFile(path) is
        this.file = new File(path)
        events.notify("open", file.name)

    method saveFile() is
        file.write()
        events.notify("save", file.name)

// Subscriber interface
interface EventListener is
    method update(filename)

class LoggingListener implements EventListener is
    private field log: File, message: string
    method update(filename) is
        log.write(replace('%s', filename, message))

class EmailAlertsListener implements EventListener is
    private field email: string, message: string
    method update(filename) is
        system.email(email, replace('%s', filename, message))

// Client configures publishers and subscribers
class Application is
    method config() is
        editor = new Editor()
        logger = new LoggingListener("/path/to/log.txt",
            "Someone opened: %s")
        editor.events.subscribe("open", logger)
        emailAlerts = new EmailAlertsListener("admin@example.com",
            "Someone changed: %s")
        editor.events.subscribe("save", emailAlerts)
Pros & Cons优缺点
  • Open/Closed Principle. Introduce new subscriber classes without changing the publisher.开闭原则。无需修改发布者代码就能引入新的订阅者类。
  • You can establish relations between objects at runtime.可以在运行时建立对象之间的联系。
  • Subscribers are notified in random order.订阅者的通知顺序是随机的。

Behavioral行为型 State

Behavior changes with state — 状态状态 / 行为随状态改变

Lets an object alter its behavior when its internal state changes. It appears as if the object changed its class. An alternative to large state-based conditionals. 当对象的内部状态改变时改变其行为，看起来就像对象改变了其类。是大型基于状态的条件语句的替代方案。

Problem & Solution问题与解决方案

The State pattern is closely related to Finite State Machines. A program can be in a limited number of states, behaving differently in each. For example, a Document can be in Draft, Moderation, or Published state — the publish() method behaves differently in each. Instead of bloated switch statements, the pattern suggests creating separate classes for each state, with the original object (context) delegating work to the current state object.

状态模式与有限状态机概念紧密相关。程序在任意时刻仅可处于几种有限的状态中，每种状态下行为不同。例如，Document可处于草稿、审阅中或已发布状态——publish()方法在每种状态下行为不同。与其使用臃肿的switch语句，不如为每种状态创建独立的类，原始对象（上下文）将工作委派给当前的状态对象。

Pseudocode伪代码

An audio player whose controls behave differently depending on the current playback state (Locked, Ready, Playing).

    method clickLock() is  state.clickLock()
    method clickPlay() is  state.clickPlay()
    method clickNext() is  state.clickNext()
    method clickPrevious() is  state.clickPrevious()

abstract class State is
    protected field player: AudioPlayer
    constructor State(player) is
        this.player = player
    abstract method clickLock()
    abstract method clickPlay()
    abstract method clickNext()
    abstract method clickPrevious()

class LockedState extends State is
    method clickLock() is
        if (player.playing)
            player.changeState(new PlayingState(player))
        else
            player.changeState(new ReadyState(player))
    method clickPlay() is  // locked, do nothing
    method clickNext() is  // locked, do nothing
    method clickPrevious() is  // locked, do nothing

class ReadyState extends State is
    method clickLock() is
        player.changeState(new LockedState(player))
    method clickPlay() is
        player.startPlayback()
        player.changeState(new PlayingState(player))
    method clickNext() is  player.nextSong()
    method clickPrevious() is  player.previousSong()

class PlayingState extends State is
    method clickLock() is
        player.changeState(new LockedState(player))
    method clickPlay() is
        player.stopPlayback()
        player.changeState(new ReadyState(player))
    method clickNext() is  player.nextSong()
    method clickPrevious() is  player.previousSong()
Pros & Cons优缺点
  • Single Responsibility. Organize state-specific code into separate classes.单一职责原则。将特定于状态的代码组织到独立的类中。
  • Open/Closed. Introduce new states without changing existing state classes or context.开闭原则。无需修改现有状态类或上下文就能引入新状态。
  • Simplify context code by eliminating bulky state machine conditionals.通过消除臃肿的状态机条件语句来简化上下文代码。
  • Can be overkill if a state machine has only a few states or rarely changes.如果状态机只有很少的状态或很少变化，应用该模式可能矫枉过正。

Behavioral行为型 Strategy

Interchangeable algorithms — 策略策略 / 可互换算法

Defines a family of algorithms, encapsulates each one, and makes them interchangeable. Lets the algorithm vary independently from clients that use it. 定义一系列算法，将每个算法封装起来，并使它们可以互换。让算法独立于使用它的客户端而变化。

Problem & Solution问题与解决方案

Imagine building a navigation app. Initially it only supports driving routes. Then you add walking, public transport, cycling, sightseeing… Each new algorithm doubles the main class size, and any change risks breaking existing code. Strategy suggests extracting each algorithm into its own class. The context holds a reference to a strategy object and delegates work to it via a common interface, letting you swap algorithms at runtime.

想象开发一款导航应用。最初只支持驾车路线，后来添加步行、公交、骑行、观光……每增加一个算法，主类体积就翻倍，任何修改都有破坏已有代码的风险。策略模式建议将每个算法提取到独立的类中。上下文持有策略对象的引用，并通过通用接口将工作委派给它，从而可在运行时替换算法。

Pseudocode伪代码

The context uses multiple strategies to execute different arithmetic operations.

上下文使用多个策略来执行不同的计算操作。

interface Strategy is
    method execute(a, b)

class ConcreteStrategyAdd implements Strategy is
    method execute(a, b) is
        return a + b

class ConcreteStrategySubtract implements Strategy is
    method execute(a, b) is
        return a - b

class ConcreteStrategyMultiply implements Strategy is
    method execute(a, b) is
        return a * b

// Context doesn't know which strategy it uses
class Context is
    private strategy: Strategy

    method setStrategy(Strategy strategy) is
        this.strategy = strategy

    method executeStrategy(a, b) is
        return strategy.execute(a, b)

// Client picks and passes the strategy
class ExampleApplication is
    method main() is
        // Create context, read numbers & action from user
        if (action == addition) then
            context.setStrategy(new ConcreteStrategyAdd())
        if (action == subtraction) then
            context.setStrategy(new ConcreteStrategySubtract())
        if (action == multiplication) then
            context.setStrategy(new ConcreteStrategyMultiply())

        result = context.executeStrategy(firstNumber, secondNumber)
Pros & Cons优缺点
  • You can swap algorithms inside an object at runtime.可以在运行时切换对象内的算法。
  • You can isolate implementation details of an algorithm from the code that uses it.可以将算法的实现细节与使用代码隔离开来。
  • You can replace inheritance with composition.可以使用组合来代替继承。
  • Open/Closed Principle. Introduce new strategies without modifying the context.开闭原则。无需修改上下文就能引入新的策略。
  • If algorithms rarely change, no reason to overcomplicate with extra classes & interfaces.如果算法极少发生改变，引入新的类和接口会让程序过于复杂。
  • Clients must be aware of the differences between strategies to pick the right one.客户端必须知晓策略间的不同才能选择合适的策略。

Behavioral行为型 Template Method

Algorithm skeleton — 模板方法模板方法 / 算法骨架

Defines the skeleton of an algorithm in a base class, letting subclasses override specific steps without changing the algorithm's overall structure. 在基类中定义算法的骨架，允许子类在不改变算法整体结构的情况下重写特定步骤。

Problem & Solution问题与解决方案

Imagine building a data-mining app that processes DOC, CSV, and PDF files. The three parsing classes contain much duplicate code — data processing and analysis logic is nearly identical, only the file-specific parts differ. Template Method suggests decomposing the algorithm into steps, making some abstract (subclasses must implement) and others with default implementations. A “hook” is an optional step with an empty body — subclasses can override it for extra extension points.

假设你要开发一款处理DOC、CSV和PDF文件的数据挖掘应用。三个解析类包含大量重复代码——数据处理和分析逻辑几乎相同，只有文件特定部分不同。模板方法建议将算法分解为一系列步骤，某些步骤是抽象的（子类必须实现），其他步骤有默认实现。"钩子"是内容为空的可选步骤——子类可以重写它以获得额外的扩展点。

Pseudocode伪代码

A strategy game AI framework where different races (Orcs, Monsters) override specific steps of the game turn algorithm.

一款策略游戏AI框架，不同种族（兽人、怪物）重写游戏回合算法的特定步骤。

Pros & Cons优缺点
  • You can let clients override only certain parts of a large algorithm, making them less affected by changes to other parts.可仅允许客户端重写大型算法的特定部分，使算法其他部分修改对其影响减小。
  • You can pull duplicate code into a superclass.可将重复代码提取到超类中。
  • Some clients may be limited by the provided skeleton of the algorithm.部分客户端可能会受到算法框架的限制。
  • Suppressing a default step via subclass may violate Liskov Substitution Principle.通过子类抑制默认步骤实现可能会违反里氏替换原则。
  • The more steps, the harder it is to maintain.模板方法中的步骤越多，维护工作越困难。

Behavioral行为型 Visitor

Separate algorithms from objects — 访问者访问者 / 分离算法与对象

Separates algorithms from the objects on which they operate by moving each algorithm into a separate class called a visitor. Lets you add new operations without modifying existing classes. 将算法与其所作用的对象分离，把每个算法移入称为"访问者"的独立类中。让你在不修改现有类的情况下添加新操作。

Problem & Solution问题与解决方案

You have a geographic graph with City, Industry, SightSeeing nodes. You need to export it to XML, but the architect refuses to modify stable node classes. Visitor places the new behavior into a separate class. Using double dispatch, each element “accepts” a visitor and calls the correct visitor method on itself — node.accept(visitor) internally calls visitor.visitCity(this). This lets you add new operations (XML export, statistics, etc.) without touching the element hierarchy.

你有一个包含城市、工业区、旅游景点节点的地理图像。你需要将其导出为XML，但架构师拒绝修改稳定的节点类。访问者模式将新行为放入独立的类中。通过双分派技巧，每个元素"接收"一个访问者并在自身上调用正确的访问者方法——node.accept(visitor)内部调用visitor.visitCity(this)。这让你在不修改元素层次结构的情况下添加新操作（XML导出、统计等）。

Pseudocode伪代码

Adding XML export support to a geometric shape hierarchy using double dispatch.

使用双分派技巧为几何图形层次结构添加XML导出支持。

// Element interface declares an accept method
interface Shape is
    method move(x, y)
    method draw()
    method accept(v: Visitor)

class Dot implements Shape is
    method accept(v: Visitor) is
        v.visitDot(this)

class Circle implements Shape is
    method accept(v: Visitor) is
        v.visitCircle(this)

class Rectangle implements Shape is
    method accept(v: Visitor) is
        v.visitRectangle(this)

class CompoundShape implements Shape is
    method accept(v: Visitor) is
        v.visitCompoundShape(this)

// Visitor interface with a method per element type
interface Visitor is
    method visitDot(d: Dot)
    method visitCircle(c: Circle)
    method visitRectangle(r: Rectangle)
    method visitCompoundShape(cs: CompoundShape)

// Concrete visitor implements the operation for all element types
class XMLExportVisitor implements Visitor is
    method visitDot(d: Dot) is
        // Export dot's ID and center coordinates
    method visitCircle(c: Circle) is
        // Export circle's ID, center, and radius
    method visitRectangle(r: Rectangle) is
        // Export rectangle's ID, top-left, width, height
    method visitCompoundShape(cs: CompoundShape) is
        // Export shape's ID and list of children IDs

// Client code runs visitor across all shapes
class Application is
    field allShapes: array of Shapes
    method export() is
        exportVisitor = new XMLExportVisitor()
        foreach (shape in allShapes) do
            shape.accept(exportVisitor)
Pros & Cons优缺点
  • Open/Closed Principle. Introduce new behaviors for different classes without modifying them.开闭原则。可以引入在不同类对象上执行的新行为，且无需修改这些类。
  • Single Responsibility. Move multiple versions of the same behavior into one class.单一职责原则。可将同一行为的不同版本移到同一个类中。
  • A visitor can accumulate useful info while working with various objects (e.g., traversing a tree).访问者对象可在与各种对象交互时收集有用信息（例如遍历对象树）。
  • You need to update all visitors each time a class is added to or removed from the element hierarchy.每次在元素层次结构中添加或移除一个类时，都要更新所有的访问者。
  • Visitors might lack access to private fields and methods of the elements they work with.访问者可能没有访问元素私有成员变量和方法的必要权限。

Note: Patterns are often used together. For example, many designs start with Factory Method and evolve toward Abstract Factory, Prototype, or Builder. Strategy can be combined with Bridge, and State can be considered a special case of Strategy. 注意：模式常常组合使用。例如，许多设计从工厂方法开始，逐步演变为抽象工厂、原型或生成器。策略可以与桥接组合使用，而状态可以被视为策略的特殊情况。
